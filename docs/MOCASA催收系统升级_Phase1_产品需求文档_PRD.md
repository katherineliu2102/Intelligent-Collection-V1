## 1. 业务背景与问题陈述

### 1.1 业务背景

MOCASA 当前菲律宾催收体系由三套独立系统支撑：

- **信贷主系统**：负责到期前 D-3 ~ D0 的 Push/SMS 提醒通知
- **中央催收系统（collection_rebuild）**：管理逾期后自动外呼（AI Call / TTS / SMS）
- **菲律宾本地 Issabel 系统**：供催收员手动拨打

三套系统数据完全不互通，无法形成用户从 D-3（到期前）到逾期后催收的完整触达轨迹，也无法形成数据闭环支撑策略优化。

### 1.2 核心业务问题

| # | 问题 | 业务影响 | 严重度 |
|---|---|---|---|
| 1 | **系统碎片化**：三套独立系统无法共享数据 | 无法追踪用户从 D-3 到 D+30 的完整催收轨迹，策略优化缺乏数据基础 | Critical |
| 2 | **Issabel 数据断层**：手动拨打记录不回流中央系统 | 大量有效催收行为数据丢失，质检存在盲区，坐席需在两套系统间切换 | Critical |
| 3 | **触达渠道单一**：仅有电话+短信，缺少 Viber/Email/WhatsApp | 接通率低（菲律宾侧审计报告确认），触达效果受限 | High |
| 4 | **分案策略过度简化**：仅按逾期天数分档 | 缺乏差异化策略，高价值用户与高风险用户使用相同触达方式 | High |
| 5 | **话术管理缺失**：无系统化话术管理，依赖个人经验 | 催收质量不稳定，话术差异大，合规风险高 | High |
| 6 | **录音资产未利用**：纯音频文件，未转写为文本 | 语音分析能力为零，无法支撑质检，Phase 2 LLM 训练无语料 | Medium |
| 7 | **用户画像不完整**：缺少行为特征、渠道偏好、历史触达汇总 | 无法支撑智能决策，Phase 2 LLM 无高质量输入 | Medium |
| 8 | **SQL 引擎规则无治理**：无版本控制、回滚、审计能力 | 规则误操作风险高，变更不可追溯 | Medium |

### 1.3 现有系统资产（保留基础）

以下资产经技术审计确认有价值，Phase 1 在其基础上升级而非重建：

| 资产 | 保留理由 |
|---|---|
| GCP PubSub 消息接入架构 | 与上游信贷系统对接方式成熟稳定 |
| SQL 引擎规则框架 | 支持免发版调整业务逻辑；增加版本治理能力后继续使用 |
| 案件全生命周期数据模型（80+ 张表） | 渐进扩展而非重建，保护历史数据 |
| XXL-Job 调度体系 | 运行稳定，复用为触达计划步骤调度器 |
| LTH 渠道集成（TTS + 短信 + AI 外呼） | 三通道已打通，作为主供应商接入统一渠道适配层 |
| MySQL + Redis 基础设施 | 保留，升级使用方式 |

## 2. 业务目标与成功指标

### 2.1 Phase 1 产品目标

将分散的到期前通知、中央催收系统、本地人工系统整合为**统一的自动化触达平台**，实现：

- **全渠道统一管理**：8 个触达渠道（自动 + 人工）统一编排与监控
- **全阶段数据闭环**：D-3 到 D+30+ 的完整触达记录与结果追踪
- **可配置差异化策略**：无需发版即可调整触达计划、策略规则、话术模板
- **Phase 2 接口预留**：标准 DecisionEngine 接口，为 LLM 智能化预留替换点

### 2.2 Phase 2 远景目标

在 Phase 1 统一平台基础上引入 LLM Agent，实现：

- Agent 自动判断案件分配的话术组
- 智能渠道选择与触达时间个性化优化
- 通话内容自动质检与改进建议
- 动态触达计划（基于用户实时反应调整后续步骤）

### 2.3 成功指标（KPI）

❓ **以下 KPI 目标值待业务负责人确认**，当前基线数据需从运营数据中提取：

| 指标类别 | 指标 | 当前基线 | Phase 1 目标 | 说明 |
|---|---|---|---|---|
| **触达效率** | 整体触达接通率（电话类渠道） | ❓ 待确认 | ❓ 提升 X% | 多渠道并发后综合接通率 |
| **触达效率** | 坐席有效通话率（接通/拨出比） | ❓ 待确认 | ❓ 提升 Y% | 预测式外呼升级后，解决坐席空等 |
| **催收效果** | S1-S2 阶段（D+1~D+10）还款转化率 | ❓ 待确认 | ❓ | 早期全渠道触达效果 |
| **催收效果** | S3-S4 阶段（D+11~D+30）案件回收率 | ❓ 待确认 | ❓ | |
| **数据完整性** | 坐席通话记录回流率 | ~0%（Issabel 数据不回流） | 100% | Issabel 废弃后全量记录 |
| **合规** | 违规触达拦截率 | ❓ 无当前数据 | 100%（所有违规触达由系统自动拦截） | |

## 3. 目标用户与使用场景

### 3.1 用户角色

| 角色 | 描述 | 主要使用系统 |
|---|---|---|
| **催收员** | 负责人工外呼和跟进录入，主要处理 S2-S4+ 阶段案件 | 催收员 App（collection-user） |
| **催收主管** | 负责监控催收进度、查看效果分析、管理坐席绩效 | 管理后台 |
| **策略配置员** | 负责配置触达计划、策略规则、话术模板 | 管理后台 |
| **系统管理员** | 负责系统配置、渠道开关管理、监控告警 | 管理后台（系统配置模块） |

### 3.2 核心使用场景

#### 场景 A：自动化全渠道触达（系统自动执行，无需人工介入）

1. 案件在 D-3 由信贷系统推送至催收系统，触达生命周期开始
2. 系统根据产品类型、用户标签、DPD 阶段匹配对应触达计划模板
3. 按模板自动执行多步骤触达（如 Push → 次日 SMS → 次日 AI Call → Viber）
4. 每步触达前进行合规校验（时段、频率），不合规则拦截并记录
5. 用户还款后，系统自动取消所有活跃触达计划；若还款事件与触达任务存在时间窗口差，触达执行前仍通过 Pre-flight Check 实时校验案件状态，已结清则终止发送

#### 场景 B：人工外呼（坐席催收，S2-S4+）

1. 系统聚合达到人工外呼阶段的案件，创建预测式外呼任务
2. LTH 平台并发拨号，接通后路由到空闲坐席
3. 坐席 App 弹屏展示：用户画像（号码脱敏）、完整触达历史、可用减免方案、建议话术
4. 通话结束后，坐席录入跟进结果（跟进标签 / 备注 / PTP 承诺时间）
5. 通话记录自动写入统一触达时间线，与自动渠道记录在同一视图呈现
6. PTP 管理（现有能力升级）：坐席可查看待跟进 PTP 列表（按承诺还款日排序），到期未履约自动标记并回传策略引擎；现有 `t_collection_ptp_info` 数据模型保留，新增列表展示和策略联动
7. 快速催收队列（现有能力升级）：从原"坐席手动选案拨打"定位升级为"待跟进提醒面板"——系统通过预测式外呼自动拨号，队列作为坐席查看待关注案件的信息视图；现有优先级排序逻辑（PTP 过期 > CBR 回访 > NOA 回访 > 最少触达）保留，改为可配置规则

#### 场景 C：策略配置与调优（策略配置员）

1. 在管理后台查看触达效果分析看板（按渠道/阶段/模板维度）
2. 发现某渠道某阶段响应率低，调整触达计划模板或更换话术版本
3. 更改立即生效，无需发版，系统实时加载新配置

#### 场景 D：用户投诉冻结触达

1. 用户通过客服渠道投诉催收行为
2. 系统对该用户所有活跃触达计划执行冻结，立即停止一切自动和人工触达
3. 冻结记录写入合规日志，待合规确认后决定是否解冻

## 4. 产品能力范围

### 4.1 Phase 1 In-scope

**P0 — Must Have（上线必须交付）**

- 统一案件生命周期管理（从 D-3 起由催收系统接管）
- 8 渠道触达适配层（Push、SMS、AI Call、TTS、Email、Viber、WhatsApp + Human Call）
- 触达计划编排引擎（可配置多步骤、多条件、多阶段）
- 合规引擎（可配置时段/频率规则，自动拦截）
- 多维度策略引擎（DPD × 产品 × 用户标签 × 金额）
- 话术模板管理（Tagalog + English 双语，版本管理，A/B 测试标记）
- 用户画像框架（建表+接口，维度数据渐进填充）
- 统一管理后台（策略/渠道/话术/监控/系统配置）
- 到期前通知接管（从信贷主系统迁移发送职责）
- 减免策略规则引擎（可配置规则，嵌入话术变量，无审批流）

**P1 — Should Have（争取上线交付）**

- 触达效果分析看板（按渠道/阶段/模板维度）
- SQL 引擎规则治理（版本管理/审计日志/沙箱测试/回滚）
- 渠道健康监控（实时可用性/延迟/成功率监控+告警）

### 4.2 Phase 1 Out-of-scope（Phase 2 实现）

| 能力 | 说明 |
|---|---|
| LLM 决策引擎 | Phase 1 预留 DecisionEngine 接口，使用规则引擎实现 |
| 通话录音 ASR 转写 | Phase 1 预留接口和表结构，供应商确定后接入 |
| 动态触达计划 | Phase 1 为固定模板计划；LLM 动态调整为 Phase 2 能力 |
| 人工审批减免流程 | Phase 1 直接规则引擎执行，无审批流 |

## 5. 渠道策略

### 5.1 全渠道触达矩阵

| 渠道 | 供应商/协议 | 触达方式 | 关键能力 | Phase 1 状态 |
|---|---|---|---|---|
| Push 通知 | FCM / APNs | App 内推送 | 零成本、即时触达、需用户安装 App | **Phase 1 接入** |
| SMS 短信 | LTH + 备选供应商 | 文本短信 | 覆盖率最高、成本适中、支持多语言模板 | **Phase 1 升级** |
| AI Robot Call | LTH TTS / 新 AI 供应商 | 自动语音播报 | 无需坐席、支持 Tagalog/English、可定制话术 | **Phase 1 升级** |
| Email | SES / SendGrid | 邮件 | 适合正式通知（欢迎函 / 催告函 / Demand Letter） | **Phase 1 接入** |
| Viber | Viber Business Messages API | 即时消息（富文本） | 菲律宾渗透率最高、富文本+按钮+图片、跨 iOS/Android | **Phase 1 接入（优先）** |
| WhatsApp | WSCRM / WhatsApp Business API | 即时消息 | 双向对话能力、用户回复率高 | **Phase 1 升级** |
| TTS 语音通知 | LTH | 自动语音播报 | 无需坐席、全自动播报还款提醒 | **Phase 1 升级** |
| Human Call 人工外呼 | LTH 预测式外呼 | 坐席通话 | 预测式并发拨号提升坐席利用率；实时弹屏展示用户画像；通话录音；PTP 管理 | **Phase 1 升级** |

> 各渠道在不同催收阶段的具体使用由策略执行层配置决定，PRD 仅保证渠道适配层接口开发计划。

### 5.2 关键渠道选型说明

**Issabel/EyeBeam 废弃**：菲律宾本地 Issabel 手动拨打系统 Phase 1 废弃。原因：通话数据不回流中央系统、数据断层无法质检、坐席需在两套系统间切换。全部功能由 LTH 预测式外呼替代。

**预测式外呼升级**：当前渐进式外呼下，大量号码无法接通，坐席空等严重（审计报告确认）。Phase 1 升级为 LTH 预测式外呼（Predictive Dialing）：系统同时拨打多路号码，接通后立即路由到空闲坐席，大幅缩短坐席等待时间。

## 6. 功能需求清单

### 6.1 P0 — Must Have

| 编号 | 功能模块 | 需求描述 | 验收标准 |
|---|---|---|---|
| F1 | 统一案件生命周期 | 案件从 D-3 起由催收系统接管。到期前通知的发送由新系统负责，不再依赖信贷主系统。到期前与到期后触达行为统一记录在同一条案件时间线上。 | 任一案件可查看从 D-3 到结清的完整触达记录 |
| F2 | 全渠道触达适配层 | 抽象统一的 ChannelAdapter 接口，Phase 1 适配 8 个渠道：SMS、Push、AI Call、TTS、Email、Viber、WhatsApp（自动触达）+ Human Call（人工触达）。每个渠道独立可启用/禁用。 | 新增渠道仅需实现接口+配置，不改核心代码；后台可独立开关每个渠道 |
| F3 | 触达计划编排引擎 | 支持按阶段（S0-S4）配置多步触达计划：每步指定渠道、模板、触发时间、前置条件（如前一步未响应）、频率上限。计划可按产品/用户标签差异化配置。所有自动触达任务在调用渠道接口前执行 Pre-flight Check：实时校验案件最新状态（Redis 缓存），已结清则立即终止发送，不依赖 XXL-Job 定时取消的时间窗口。 | 通过后台配置即可创建/修改触达计划，无需发版；已结清案件在发送前被 Pre-flight Check 拦截，不发生误触达 |
| F4 | 合规引擎 | 可配置合规规则引擎：支持触达时段、频率上限等规则配置。所有触达行为前置合规校验，不合规则自动拦截并记录。规则阈值通过管理后台设定，无需硬编码。 | 违反合规规则的触达请求被自动拦截并记录；规则可通过后台配置调整 |
| F5 | 策略引擎升级 | 分案/触达策略从单一逾期天数升级为多维条件组合：DPD 区间 × 产品类型 × 用户标签 × 金额区间 × 历史行为。支持规则优先级和 fallback。Phase 1 上线时最小核心标签集必须就位（见 F7）。 | 可配置 20+ 条差异化策略规则，命中率可追踪；最小核心标签集可用于策略命中 |
| F6 | 话术模板管理 | 系统化管理各阶段、各渠道的话术模板（Tagalog + English 双语）。支持变量替换（姓名/金额/日期）、版本管理、A/B 测试标记。 | 后台可维护全部话术模板；模板变量可从用户画像自动填充 |
| F7 | 完整用户画像 | 整合现有 8 张用户表，新增扩展表。Phase 1 建立框架+接口，维度数据从数仓渐进填充。所有维度通过 ProfileService 统一对外提供。**Phase 1 最小核心标签集（必须随主系统同步交付）**：历史还款频次、历史逾期次数、PTP 履约率、历史投诉标记、首选触达渠道。 | 单接口返回用户画像 JSON；最小核心标签集上线日可用；其余字段可渐进补全 |
| F8 | 统一管理后台 | 整合策略配置、渠道管理（含开关控制）、话术管理、触达计划、案件监控（含案件详情查看）、系统配置到一个管理后台。 | 所有配置操作通过管理后台完成，无需直接操作数据库 |
| F9 | 到期前通知接管 | 到期前 Push/SMS 的发送职责从信贷主系统迁移至新催收系统。信贷主系统仅推送案件数据（D-3 起），新系统负责触达决策和发送。历史发送记录通过数仓 ETL 全量迁移 + PubSub 增量同步实现过渡。 | D-3 起的通知由催收系统独立发送，信贷主系统不再直接触达用户 |
| F10 | 减免策略规则引擎 | 将现有减免方案（100% 罚息减免 / 25% 本金折扣等）整理为规则配置：按 DPD 阶段 × 产品编号 × 用户标签匹配减免方案，嵌入话术模板变量自动填充。无人工审批流。内置熔断机制：当某条减免规则在单位时间内触发量异常放大时，系统自动挂起该规则并告警。 | 后台可配置减免规则（含产品维度）；话术模板自动引用对应减免方案；熔断触发后规则自动挂起 |

### 6.2 P1 — Should Have

| 编号 | 功能模块 | 需求描述 |
|---|---|---|
| F11 | 触达效果分析看板 | 按渠道/阶段/模板维度统计触达率、响应率、转化率（触达→还款），支撑策略配置员优化策略 |
| F12 | SQL 引擎规则治理 | 为现有 SQL 引擎增加版本管理、变更审计日志、沙箱测试、一键回滚能力 |
| F13 | 渠道健康监控 | 实时监控各渠道供应商的可用性、延迟、成功率，异常时自动告警并触发降级策略 |

### 6.3 P2 — Phase 2 预留

Phase 1 仅建接口/表结构，Phase 2 实现完整功能：

| 编号 | 功能模块 | Phase 1 预留 | Phase 2 实现 |
|---|---|---|---|
| F14 | LLM 决策接口 | DecisionEngine 统一接口，Phase 1 用 RuleBasedEngine | LLMAgentEngine 替换，支持话术组推荐、触达编排优化 |
| F15 | 决策日志与可解释性 | `t_decision_log` 表，记录规则命中 | 记录 LLM 推理过程（Chain of Thought） |
| F16 | 通话录音 ASR 转写 | 预留 TranscriptionService 接口 + `t_call_transcript` 表 | LLM 分析通话转写文本，自动评分+改进建议 |
| F17 | 动态触达计划 | 固定模板触达计划 | LLM Agent 根据用户实时反应动态调整后续步骤 |

## 7. 监管与合规要求

本章节描述产品必须满足的监管约束，是合规引擎的需求来源。系统实现方案见[架构设计文档 §1.4](./MOCASA催收系统升级_Phase1_架构设计文档.md)。

### 7.1 适用法规

- **SEC Memorandum Circular No. 18-2019**（菲律宾贷款公司催收行为规范，核心约束来源）
- **R.A. 3765**（Truth in Lending Act）
- **R.A. 10173**（Data Privacy Act of 2012）

### 7.2 Phase 1 合规约束（基础版）

Phase 1 合规要求以系统可配置为目标，初期规则宽松，后续通过管理后台按需收紧。

| 合规项 | Phase 1 实现 |
|---|---|
| **触达时段** | 可配置允许触达时段（默认 6:00 AM ~ 10:00 PM），通过管理后台调整 |
| **触达频率上限** | 可配置单用户单日触达上限（默认值由策略配置员设定），按用户维度统计 |
| **预测式外呼呼损率** | 可配置呼损率上限，超阈值自动降级为渐进式外呼，防止接通后无坐席秒挂引发投诉 |
| **隐私保护** | 坐席端号码全部脱敏；不向非借款人发送催收信息 |

> Phase 1 合规引擎核心能力：规则可配置 + 违规自动拦截 + 拦截记录可查。具体阈值由策略配置员按业务需要设定，不硬编码。

### 7.3 合规产品行为要求

- 系统在触达行为前自动完成合规校验
- 违规触达请求自动拦截并记录
- 合规规则通过管理后台配置，无需发版修改

## 8. 非功能要求

❓ **本章节待技术负责人和业务方共同确认具体指标**，以下为基于业务场景的建议方向：

### 8.1 可用性与性能

| 要求 | 建议指标 | 说明 |
|---|---|---|
| 系统可用性 | ❓ ≥ 99.5% | 催收系统为业务关键系统，但非实时金融交易系统 |
| 触达计划执行延迟 | ❓ 计划触达时间 ±5 分钟内 | 涉及合规时段控制 |
| 管理后台响应时间 | ❓ P99 < 3 秒 | |
| 还款事件响应延迟 | ≤ 5 分钟（取消活跃触达计划） | 主要瓶颈在支付渠道确认，技术链路约 2-5 分钟；触达执行前 Pre-flight Check（F3）作为兜底，避免取消延迟窗口内误发送 |

### 8.2 数据保留

| 数据类型 | 建议保留周期 | 说明 |
|---|---|---|
| 触达时间线记录 | ❓ 至少 3 年 | 合规审计需要 |
| 通话录音 | ❓ 至少 1 年 | |
| 合规违规记录 | ❓ 至少 5 年 | 建议与监管要求对齐，待法务确认 |
| 决策日志 | ❓ 至少 1 年 | Phase 2 LLM 训练数据来源 |

### 8.3 语言支持

- 系统管理后台：英文界面
- 话术模板：Tagalog + English 双语（所有渠道全覆盖）
- 通话录音 ASR：需支持 Tagalog / Taglish（Phase 2 接入时的硬性要求）

### 8.4 隐私与数据安全

- 敏感字段（手机号、证件号）在所有展示层统一脱敏，坐席不可查看借款人或联系人的完整号码
- 系统不向非借款人（含紧急联系人）发送任何催收信息
- 第三方渠道供应商 API 凭证通过 `credentials_ref` 间接引用，不明文存储
- 催收员只能查看被分配的案件数据，不可跨权限访问

❓ **以下安全要求待确认**：

- 管理后台的角色权限体系（RBAC）细化规则（如主管是否可查看所有催收员的案件）
- 是否需要管理后台操作审计日志（谁在什么时间修改了哪条规则）
- 数据出境是否有监管限制（用户数据是否可传输至 LTH 等境外供应商）

## 9. 关键产品决策记录

基于产品负责人确认（2026-05-12），以下决策已锁定：

| 决策项 | 结论 | 决策理由 |
|---|---|---|
| 到期前通知接管 | **新系统接管发送** | 到期前 Push/SMS 的发送职责从信贷主系统迁移至新催收系统。案件从 D-3 起由催收系统统一管理触达。信贷主系统仅负责案件数据推送，不再直接发送通知。历史发送记录全量迁移。 |
| 用户画像细化节奏 | Phase 1 建框架，数据从数仓渐进填充 | 画像扩展表结构 Phase 1 就位，维度数据后期从已有数仓逐步导入，不阻塞主流程。 |
| 减免策略 | Phase 1 规则引擎执行，**无人工审批流** | 将现有减免方案（100% 罚息减免 / 25% 本金折扣等）整理为可配置规则，嵌入话术模板变量。不做审批流。 |
| 录音 ASR 转写 | 供应商接触中，Phase 1 预留接口 | ASR 需支持 Tagalog/Taglish，供应商尚未确定。Phase 1 预留 TranscriptionService 接口和 `t_call_transcript` 表，供应商确定后接入。 |
| 工作流引擎 | **Phase 1 不引入 Flowable** | 触达计划用自建状态机 + XXL-Job 调度，Phase 1 需求可控，状态机足够。Phase 2 如 LLM 需要动态修改流程再评估 Flowable。 |
| 案件分配方式 | **全部由系统自动分配** | 案件分配逻辑统一由策略引擎根据 DPD × 产品 × 用户标签等维度自动决策，不设人工手动分配入口，确保分配一致性和可追溯性。 |
| Skip Tracing 范围 | **仅展示脱敏联系人信息，不触达非借款人** | 坐席 App 可查看借款人紧急联系人等信息（号码脱敏），但系统不向非借款人发送任何催收信息（含自动渠道和人工渠道）。 |
| PTP 到期未履约处理 | **作为策略矩阵输入数据** | PTP 到期未还款为策略引擎输入维度之一，系统自动将该信号传导至策略与执行层，由策略规则决定后续触达强度和渠道升档。 |
| 用户投诉处理 | **冻结该用户全部触达** | 用户投诉后系统对该用户采取触达冻结措施，冻结期间所有自动和人工触达均暂停。 |
| 催收员 App 功能优先级 | **PTP 管理 > 快速催收队列 > 绩效统计** | PTP 管理和快速催收队列为 Phase 1 P0 交付；绩效统计排序 Phase 1 仅完成底层数据收集，展示功能推迟。 |
| 坐席号码可见性 | **全部脱敏，坐席不可见完整号码** | 借款人与联系人号码在坐席 App 中全部脱敏展示，不影响系统后台发起渠道触达。 |
| 绩效报表导出 | **不在催收系统中实现** | 所有催收数据回流至信贷主系统，催收主管自行从数据库取数计算绩效。催收系统仅负责底层数据记录。 |
| 新坐席入职权限 | **管理后台自建用户管理** | 系统管理员在管理后台完成坐席账号创建、角色分配、LTH 坐席组关联。离职时禁用账号即停止分配。不对接外部 HR 系统。 |
| 渠道故障降级策略 | **自动降级 + 手动全局切流** | 单渠道连续报错超阈值自动降级（如 Viber 不可用自动切补发 SMS）；管理后台提供一键全局切流开关，应对供应商大面积故障或合规突发检查。 |
| 减免规则按产品差异化 | **支持，条件含 product_code** | 减免规则在 strategy_rule 中配置，条件组合包含产品编号维度，支持按产品差异化匹配不同减免方案。 |
| RBAC 权限体系 | **保留现有框架，Phase 1 不新增细粒度权限** | 复用现有 `t_system_role` / `t_system_role_function` 权限框架；角色和功能权限树按新系统模块更新；催收员 App 端维持现有 admin/催收员二元模型，Phase 1 不引入多级主管审批。 |
