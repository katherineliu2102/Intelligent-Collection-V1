## 1. 业务背景与问题陈述

### 1.1 业务背景

MOCASA 当前菲律宾催收体系由三套独立系统支撑：

- **信贷主系统**：负责到期前 D-3 ~ D0 的 Push/SMS 提醒通知
- **中央催收系统（collection_rebuild）**：管理逾期后自动外呼（AI Call / TTS / SMS）
- **菲律宾本地 Issabel 系统**：供催收员手动拨打

三套系统数据完全不互通，无法形成用户从 D-3（到期前）到逾期后催收的完整触达轨迹，也无法形成数据闭环支撑策略优化。

### 1.2 核心业务问题

| # | 问题 | 业务影响 | 严重度 |
|---|---|---|---|
| 1 | **系统碎片化**：三套独立系统无法共享数据 | 无法追踪用户从 D-3 到 D+30 的完整催收轨迹，策略优化缺乏数据基础 | Critical |
| 2 | **Issabel 数据断层**：手动拨打记录不回流中央系统 | 大量有效催收行为数据丢失，质检存在盲区，坐席需在两套系统间切换 | Critical |
| 3 | **触达渠道单一**：仅有电话+短信，缺少 Viber/Email/WhatsApp | 接通率低（菲律宾侧审计报告确认），触达效果受限 | High |
| 4 | **分案策略过度简化**：仅按逾期天数分档 | 缺乏差异化策略，高价值用户与高风险用户使用相同触达方式 | High |
| 5 | **话术管理缺失**：无系统化话术管理，依赖个人经验 | 催收质量不稳定，话术差异大，合规风险高 | High |
| 6 | **录音资产未利用**：纯音频文件，未转写为文本 | 语音分析能力为零，无法支撑质检，Phase 2 LLM 训练无语料 | Medium |
| 7 | **用户画像不完整**：缺少行为特征、渠道偏好、历史触达汇总 | 无法支撑智能决策，Phase 2 LLM 无高质量输入 | Medium |

### 1.3 现有系统资产（保留基础）

以下资产经技术审计确认有价值，Phase 1 在其基础上升级而非重建：

| 资产 | 保留理由 |
|---|---|
| GCP PubSub 消息接入架构 | 与上游信贷系统对接方式成熟稳定 |
| 规则化策略引擎（RuleBasedDecisionEngine） | 触达计划、合规规则、话术分级经 Nacos / 配置中心热更新，支持免发版调整策略 |
| 案件全生命周期数据模型（80+ 张表） | 渐进扩展而非重建，保护历史数据 |
| XXL-Job 调度体系 | 运行稳定，复用为触达计划步骤调度器 |
| LTH 渠道集成 | TTS / 人工外呼由 LTH 现网独立编排；AI 对话外呼经 AI Call 供应商；SMS/Push 经通知中心接入 |
| MySQL + Redis 基础设施 | 保留，升级使用方式 |

## 2. 业务目标与成功指标

### 2.1 Phase 1 产品目标

将分散的到期前通知与中央催收系统整合为**统一的自动化触达平台**，实现：

- **全渠道统一管理**：4 个自动触达渠道（Push / Email / SMS / AI Call）统一编排与监控
- **全阶段数据闭环**：D-3 到 D+30+ 的完整触达记录与结果追踪
- **可配置差异化策略**：无需发版即可调整触达计划、策略规则、话术模板
- **Phase 2 接口预留**：标准 DecisionEngine 接口，为 LLM 智能化预留替换点

### 2.2 Phase 2 远景目标

在 Phase 1 统一平台基础上引入 LLM Agent，实现：

- Agent 自动判断案件分配的话术组
- 智能渠道选择与触达时间个性化优化
- 通话内容自动质检与改进建议
- 动态触达计划（基于用户实时反应调整后续步骤）

### 2.3 成功指标（KPI）

Phase 1 以**触达率**与**回收率**为核心关注指标：通过统一全渠道编排与差异化策略，提升用户触达效率；通过全阶段数据闭环与策略可配置，改善各逾期阶段的还款回收效果。基线来源：[LTH 路由诊断报告](../../智能催收(old)/docs/history/lth_issabel_routing_analysis_report.md)（触达效率）、[历史回收率报表](../../智能催收(old)/docs/history/历史回收率.pdf)（一期，2026-03~06 均值）。Stage 口径见 [渠道编排规格](./MOCASA催收系统升级_Phase1_渠道编排规格.md)。

| 指标类别 | 指标                  | 当前基线                     |
| ---- | ------------------- | ------------------------ |
| 触达效率 | AI Call 去重用户触达率 | 均值约 3.4%（LTH 样本，2026-05） |
| 触达效率 | SMS 触达成功率           | 待补充基线                    |
| 催收效果 | S1–S2 回收率（DPD1–15）  | S1 ~36% / S2 ~17%        |
| 催收效果 | S3–S4 回收率（DPD16–90） | S3 ~5% / S4 ~4%          |
| 合规   | 违规触达拦截率             | 无历史基线（目标 100%）           |

## 3. 目标用户与使用场景

### 3.1 用户角色

> **说明**：催收员（坐席）不在本系统角色范围内——人工外呼及跟进录入仍沿用原 LTH 系统，本系统不接管催收员的作业入口。

| 角色 | 描述 | 主要使用系统 |
|---|---|---|
| **催收主管** | 负责监控催收进度、查看效果分析 | 管理后台 |
| **策略配置员** | 负责配置触达计划、策略规则、话术模板 | 管理后台 |
| **系统管理员** | 负责系统配置、渠道开关管理、监控告警 | 管理后台（系统配置模块） |

### 3.2 核心使用场景

#### 场景 A：自动化全渠道触达（系统自动执行，无需人工介入）

1. 案件在 D-3 由信贷系统推送至催收系统，触达生命周期开始
2. 系统根据产品类型、用户标签、DPD 阶段匹配对应触达计划模板
3. 按模板自动执行多步骤触达（如 Push → Email → SMS → AI Call）
4. 每步触达前进行合规校验（时段、频率），不合规则拦截并记录
5. 用户还款后，系统自动取消所有活跃触达计划；若还款事件与触达任务存在时间窗口差，触达执行前仍通过 Pre-flight Check 实时校验案件状态，已结清则终止发送

> **说明**：人工外呼（坐席催收）不在本系统范围内，仍由原 LTH 系统承载，故不列为本系统的核心使用场景。

#### 场景 B：策略配置与调优（策略配置员）

1. 在管理后台查看触达效果分析看板（按渠道/阶段/模板维度）
2. 发现某渠道某阶段响应率低，调整触达计划模板或更换话术版本
3. 更改立即生效，无需发版，系统实时加载新配置

#### 场景 C：用户投诉冻结触达（Phase 2 终态取消）

> Phase 1：投诉后仅通过引擎 PreFlight 可恢复冻结（实时校验停住自动触达，不取消计划），见 [核心引擎规格 §5 ②](./MOCASA催收系统升级_Phase1_核心引擎规格.md#5-步骤执行管线)。**`COMPLAINT` / `MANUAL` 带外终态取消**（管理后台写 `cancel_reason`）为 Phase 2 能力，见 [领域模型 §2.7](./MOCASA催收系统升级_Phase1_领域模型与数据定义.md#27-cancelreason计划取消原因)。

1. 用户通过客服渠道投诉催收行为
2. 客服/后台对该用户活跃触达计划执行**可恢复冻结**（实时标记，引擎执行前实时校验即"停住"自动触达，不取消计划）；人工外呼仍在 LTH 现网，按客服 SOP 处理
3. **（Phase 2）** 冻结记录写入合规日志：确认无违规 → **解冻**（清标记后恢复触达）；确认违规 → 管理后台带外写入**终态取消**（`COMPLAINT`）不再续建

## 4. 产品能力范围

### 4.1 Phase 1 In-scope

**P0 — Must Have（上线必须交付）**

- 统一案件生命周期管理（从 D-3 起由催收系统接管）
- 4 渠道触达适配层（Push、Email、SMS、AI Call）
- 触达计划编排引擎（可配置多步骤、多条件、多阶段）
- 合规引擎（可配置时段/频率规则，自动拦截）
- 多维度策略引擎（DPD × 产品 × 用户标签 × 金额）
- 话术模板管理（Phase 1 英文，按 Stage × 渠道配置，变量替换，配置中心热更新）
- 用户画像框架（建表+接口，维度数据渐进填充）
- 统一管理后台（策略/渠道/话术/监控/系统配置）
- 到期前通知接管（D-3 ~ D0 的 Push/SMS 提醒）

**P1 — Should Have（争取上线交付）**

- 触达效果分析看板（按渠道/阶段/模板维度）
- 渠道健康监控（实时可用性/延迟/成功率监控+告警）

### 4.2 Phase 1 Out-of-scope（Phase 2 实现）

| 能力 | 说明 |
|---|---|
| LLM 决策引擎 | Phase 1 预留 DecisionEngine 接口，使用规则引擎实现 |
| 通话录音 ASR 转写 | Phase 1 预留接口和表结构，供应商确定后接入 |
| 动态触达计划 | Phase 1 为固定模板计划；LLM 动态调整为 Phase 2 能力 |
| 减免策略规则引擎 | Phase 1 暂不考虑（含人工审批减免流程）；后续阶段再评估 |
| 旧系统 SQL 引擎规则（`t_sql_engine_rule`） | 属 collection_rebuild 遗留能力（分案落库等）；新系统不沿用，不设治理需求 |

## 5. 渠道策略

Phase 1 自动触达渠道矩阵如下（各阶段具体使用由策略执行层配置决定，PRD 仅保证渠道适配层接口开发计划）：

| 渠道 | 供应商/协议 | 触达方式 | 关键能力 | Phase 1 状态 |
|---|---|---|---|---|
| Push 通知 | 通知中心 → JPush | App 内推送 | 零成本、即时触达、需用户安装 App | **Phase 1 接入** |
| Email | SendGrid（主；SES 备，Phase 1 不实现切换） | 邮件 | 适合正式通知（欢迎函 / 催告函 / Demand Letter） | **Phase 1 接入** |
| SMS 短信 | 通知中心（内部路由底层 SMS 供应商） | 文本短信 | 覆盖率最高、成本适中、支持多语言模板 | **Phase 1 接入** |
| AI Call | AI Call 供应商（对话外呼） | 可对话 AI 外呼 | 全自动、支持 Tagalog/English、可定制话术与对话分支 | **Phase 1 接入** |
| Viber | Viber Business Messages API | 即时消息（富文本） | 菲律宾渗透率最高、富文本+按钮+图片、跨 iOS/Android | **Phase 1 升级** |
| WhatsApp | WSCRM / WhatsApp Business API | 即时消息 | 双向对话能力、用户回复率高 | **Phase 1 升级** |

> 人工外呼（Human Call）仍由原 LTH 系统承载，不在本系统渠道矩阵内（见 §3）。

## 7. 监管与合规要求

本章节描述产品必须满足的监管约束，是合规引擎的需求来源。系统实现方案见[架构设计文档 §1.4（渠道编排层）](./MOCASA催收系统升级_Phase1_架构设计文档.md#14-渠道编排层南向出站)。

### 7.1 适用法规

- **SEC Memorandum Circular No. 18-2019**（菲律宾贷款公司催收行为规范，核心约束来源）
- **R.A. 3765**（Truth in Lending Act）
- **R.A. 10173**（Data Privacy Act of 2012）

### 7.2 Phase 1 合规约束（基础版）

Phase 1 合规要求以系统可配置为目标，初期规则宽松，后续通过管理后台按需收紧。

| 合规项 | Phase 1 实现 |
|---|---|
| **触达时段** | 可配置允许触达时段（默认 6:00 AM ~ 10:00 PM），通过管理后台调整 |
| **触达频率上限** | 可配置单用户单日触达上限（默认值由策略配置员设定），按用户维度统计 |
| **预测式外呼呼损率** | 可配置 AI Call 预测式外呼呼损率上限，超阈值自动降级为渐进式拨号，避免接通后 AI 无应答/秒挂引发投诉 |
| **隐私保护** | 展示层号码全部脱敏；不向非借款人发送催收信息 |

> Phase 1 合规引擎核心能力：规则可配置 + 违规自动拦截 + 拦截记录可查。具体阈值由策略配置员按业务需要设定，不硬编码。

### 7.3 合规产品行为要求

- 系统在触达行为前自动完成合规校验
- 违规触达请求自动拦截并记录
- 合规规则通过管理后台配置，无需发版修改

## 8. 非功能要求

### 8.1 语言支持

- 系统管理后台：英文界面
- 话术模板：Phase 1 英文（覆盖 Push/Email/SMS/AI Call 全渠道）；Tagalog 本地化为 Phase 2
- 通话录音 ASR：需支持 Tagalog / Taglish（Phase 2 接入时的硬性要求）

### 8.2 隐私与数据安全

- 敏感字段（手机号、证件号）在所有展示层统一脱敏，管理后台不展示借款人或联系人的完整号码
- 系统不向非借款人（含紧急联系人）发送任何催收信息
- 第三方渠道供应商 API 凭证通过 `credentials_ref` 间接引用，不明文存储
- 后台用户按角色仅可访问授权范围内的案件数据，不可跨权限访问

❓ **以下安全要求待确认**：

- 管理后台的角色权限体系（RBAC）细化规则（如主管是否可查看全部案件）
- 是否需要管理后台操作审计日志（谁在什么时间修改了哪条规则）
- 数据出境是否有监管限制（用户数据是否可传输至 LTH 等境外供应商）

## 9. 关键产品决策记录

基于产品负责人确认（2026-05-12），以下决策已锁定：

| 决策项 | 结论 | 决策理由 |
|---|---|---|
| 到期前通知接管 | **新系统接管发送** | 到期前 Push/SMS 的发送职责从信贷主系统迁移至新催收系统。案件从 D-3 起由催收系统统一管理触达。信贷主系统仅负责案件数据推送，不再直接发送通知。历史发送记录全量迁移。 |
| 用户画像细化节奏 | Phase 1 建框架，数据从数仓渐进填充 | 画像扩展表结构 Phase 1 就位，维度数据后期从已有数仓逐步导入，不阻塞主流程。 |
| 减免策略 | **Phase 1 暂不考虑** | 减免策略规则引擎（含话术变量填充、产品差异化、熔断、审批流）整体推迟至后续阶段评估，Phase 1 不交付（见 §4.2）。 |
| 录音 ASR 转写 | 供应商接触中，Phase 1 预留接口 | ASR 需支持 Tagalog/Taglish，供应商尚未确定。Phase 1 预留 TranscriptionService 接口和 `t_call_transcript` 表，供应商确定后接入。 |
| 工作流引擎 | **Phase 1 不引入 Flowable** | 触达计划用自建状态机 + XXL-Job 调度，Phase 1 需求可控，状态机足够。Phase 2 如 LLM 需要动态修改流程再评估 Flowable。 |
| 案件分配方式 | **全部由系统自动分配** | 案件分配逻辑统一由策略引擎根据 DPD × 产品 × 用户标签等维度自动决策，不设人工手动分配入口，确保分配一致性和可追溯性。 |
| Skip Tracing 范围 | **不触达非借款人** | 本系统不向非借款人（含紧急联系人）发送任何催收信息。联系人信息的人工查看由原 LTH 人工催收系统承载，本系统展示层一律脱敏。 |
| PTP 到期未履约处理 | **Phase 2 预留，Phase 1 不处理** | Phase 1 不记录 PTP、不将 PTP 到期作为策略升档输入，引擎不消费 `PTP_EXPIRED`；坐席 PTP 录入仍在 LTH 现网。Phase 2 再将 PTP 到期信号纳入策略矩阵驱动升档。 |
| 用户投诉处理 | **Phase 1 可恢复冻结；Phase 2 违规则终态取消** | Phase 1：投诉后自动触达实时冻结（"停住"、不取消计划），解冻可恢复。Phase 2：合规确认违规后管理后台带外写入 `COMPLAINT`/`MANUAL` 终态取消。人工外呼在 LTH 现网，按客服 SOP。 |
| 人工催收员能力边界 | **本系统不提供坐席外呼，仅提供查看/配置** | 人工外呼、PTP 跟进录入、快速催收队列等坐席作业仍在原 LTH 系统；本系统仅向催收主管/策略配置员提供效果、进度查看与策略/话术/渠道配置。 |
| 数据展示脱敏 | **借款人/联系人号码全部脱敏** | 本系统所有展示层（看板、案件详情）号码一律脱敏，不影响系统后台自动渠道触达。 |
| 绩效报表导出 | **不在催收系统中实现** | 所有催收数据回流至信贷主系统，催收主管自行从数据库取数计算绩效。催收系统仅负责底层数据记录。 |
| 后台账号管理 | **管理后台自建用户管理** | 系统管理员在管理后台创建主管/策略配置员/管理员等后台账号并分配角色，离职时禁用账号。不含人工催收员坐席账号（坐席仍在原 LTH 系统管理），不对接外部 HR 系统。 |
| 渠道故障降级策略 | **自动降级 + 手动全局切流** | 单渠道连续报错超阈值自动降级（如 Viber 不可用自动切补发 SMS）；管理后台提供一键全局切流开关，应对供应商大面积故障或合规突发检查。 |
| RBAC 权限体系 | **保留现有框架，Phase 1 不新增细粒度权限** | 复用现有 `t_system_role` / `t_system_role_function` 权限框架；角色与功能权限树按新系统后台模块（催收主管 / 策略配置员 / 系统管理员）更新；Phase 1 不引入多级主管审批。 |
