### 1.2 分层架构与核心引擎设计

#### 1.2.1 分层映射

| 层 | 职责 | 边界约束 | 代码模块 |
|---|---|---|---|
| 数据接入层 | 接收上游推送，清洗/入库，生成 context_snapshot，发布领域事件 | 不做业务决策；不直接调用渠道 | `collection-ingestion` |
| 核心引擎 | 事件消费、状态机、步骤执行骨架、并发控制、故障恢复 | 不直接调用供应商 API；不包含业务规则 | `engine.lifecycle` + `engine.spi` |
| 渠道编排 | 业务决策（策略子层）+ 渠道执行（执行子层） | 执行子层不查业务数据库；策略子层决策基于 snapshot | `engine.strategy` + `collection-channel` |
| 数据服务层 | 核心业务数据管理 | 纯数据存取，不含编排/触发逻辑 | `collection-service` |
| 应用层 | 对外 API；Webhook 入站路由；XXL-Job 触发器 | 不含业务逻辑 | `collection-admin` |

共享基础模块 `collection-common`：DTO / 枚举 / 工具 / 异常 / `CollectionEventBus` 接口 / SPI 接口定义。

> 保持单体应用部署，模块间通过 Service 接口通信，不做微服务拆分，但模块边界清晰到可独立编译。

#### 1.2.2 步骤执行骨架

`StepExecutionOrchestrator` 定义步骤从触发到完成的固定流程——只关心串联顺序，不关心每个环节的具体业务逻辑：

```
executeStep(plan, step):

  ① 幂等锁获取 ──── 重复事件 → 静默退出
  ② 系统级守卫 ──── 案件已还款/冻结/关闭 → 静默退出
  ③ 业务级守卫 ──── SPI: ExecutionGuard → 合规拦截 → SKIPPED → 推进
  ④ 步骤解析   ──── SPI: StepResolver → 输出 StepCommand
  ⑤ 渠道调度   ──── ChannelGateway.dispatch(command) → StepResult
  ⑥ 结果处理   ──── 失败+可重试 → 退避重试 / 失败+不可重试 → FAILED
  ⑦ 推进决策   ──── SPI: AdvancementPolicy → 注册下一步 / 完成 / 穷尽
```

**渠道类型对状态的影响**：

| 渠道类别 | 调用后状态 | 完成方式 |
|---|---|---|
| 消息类 (SMS/Push/Email/Viber/WhatsApp) | STEP_WAITING 或直接推进 | 同步返回成功 |
| 电话类 (AI_CALL / TTS(Text-to-Speech, 文字转语音播报)) | 保持 STEP_EXECUTING | 等待 CHANNEL_CALLBACK 事件 |
| 人工类 (HUMAN_CALL) | 保持 STEP_EXECUTING | 等待坐席结果回调 |

> 完整状态机转换规格和并发竞态控制见 [渠道编排规格 §2-§4](./MOCASA催收系统升级_Phase1_渠道编排规格.md)。

#### 1.2.3 核心引擎 SPI 架构

核心引擎与渠道编排通过 **SPI（Service Provider Interface）模式**解耦：

- **核心引擎**知道"什么时候该做什么"——事件到达时驱动状态机转换，在正确的时刻调用 SPI 接口
- **渠道编排**知道"做什么决策、怎么执行"——实现 SPI 接口，回答每个决策点的业务问题

**设计动机**：

| 收益 | 说明 |
|---|---|
| 独立开发 | lifecycle 使用 Mock SPI 跑通全链路；渠道编排独立验证决策正确性 |
| 变更隔离 | 业务规则变化不影响状态机；状态机重构不影响策略逻辑 |
| Phase 2 替换 | LLM 引擎只需新增 SPI 实现类，lifecycle 零修改 |

**核心引擎组件**：

| 组件 | 职责 |
|---|---|
| `EventConsumerDispatcher` | 消费 Redis Stream 事件，路由到状态机，终态拦截，并发锁获取 |
| `PlanLifecycleManager` | 状态机（7 状态 / 2 终态），管理计划全生命周期状态转换 |
| `StepExecutionOrchestrator` | 步骤执行骨架（见 [§1.2.2](#122-步骤执行骨架)） |
| `PreFlightChecker` | 系统级守卫：案件是否已还款/冻结/关闭 |

`PreFlightChecker` 与 SPI `ExecutionGuard` 的区别：前者是系统级不变量（案件还活着吗），后者是业务级可配规则（合规频率/时段限制）。前者失败静默退出，后者失败标记 SKIPPED 并推进下一步。

#### 1.2.4 SPI 接口概要

5 个 SPI 接口覆盖计划生命周期中的全部业务决策点，**不同生命周期阶段触发不同的 SPI 子集**（单次步骤执行最多调用 3 个）：

| SPI | 调用时机 | 决策问题 | 输入 → 输出 | Phase 1 实现 |
|---|---|---|---|---|
| `PlanFactory` | 案件入库 / 阶段变更 | 创建什么触达计划？ | 案件+阶段 → 计划+步骤序列 | 规则匹配模板，实例化 |
| `ExecutionGuard` | 步骤执行前 | 这一步允许执行吗？ | 用户+渠道+时间 → 允许/拦截 | 合规频率+时段校验 |
| `StepResolver` | Guard 通过后 | 具体发什么、用什么渠道？ | 计划+步骤 → StepCommand | 渠道选择+命令组装 |
| `AdvancementPolicy` | 步骤完成后 | 下一步是什么？ | 计划+结果 → 推进/完成/穷尽 | 按模板顺序推进 |
| `ExhaustionPolicy` | 计划穷尽后 | 所有步骤用完怎么办？ | 计划+案件 → 续建/升档/完成 | 规则判定续建策略 |

> 完整的 SPI 接口定义（方法签名、DTO 字段、实现约束）见 [核心引擎规格 §4 SPI](./MOCASA催收系统升级_Phase1_核心引擎规格.md#4-spi-接口完整定义)；异常恢复见 [§5](./MOCASA催收系统升级_Phase1_核心引擎规格.md#5-异常恢复策略)。

#### 1.2.5 开发协作边界

**接口先行**：核心引擎侧先完成 SPI 接口定义 + DTO，作为编译期契约，之后并行开发：

| 阶段 | 核心引擎 | 渠道编排 |
|---|---|---|
| **契约** | 定义 SPI 接口 + StepCommand/StepResult | Review 接口，确认满足策略和渠道需求 |
| **骨架** | lifecycle 全链路（事件消费、状态机、并发控制、重试、DLQ） | 策略实现 + 渠道 Adapter |
| **集成** | 替换 Mock SPI，接入真实实现 | 端到端联调 |

**变更隔离**：

| 变更场景 | 影响范围 |
|---|---|
| 新增合规规则 / 调整频率阈值 | 仅改渠道编排（策略子层） |
| 新增渠道类型 | 仅改渠道编排（两个子层） |
| 修改状态机转换规则 | 仅改核心引擎 |
| Phase 2 LLM 替换决策 | 渠道编排新增实现类，配置切换 |

### 1.3 关键架构机制

> 本节列出支撑事件驱动架构可靠运行的关键机制——**缺少任何一个，事件驱动模型在特定场景下会失效**。纯模块内部的实现策略（如渠道熔断）不在此列，归属对应模块文档。

#### 1.3.1 事件驱动 + 定时触发

**缺失后果**：模块间直接调用，耦合度高；调度线程被渠道 I/O 阻塞。

系统采用**事件驱动为主、定时触发为辅**的混合模式。8 种领域事件通过 Redis Stream 流转；XXL-Job 采用 **Trigger-to-Event** 模式——仅发布事件立即返回，业务逻辑全部由 Consumer 线程池执行。两个上下文严格隔离，渠道供应商变慢只影响 Consumer 池，不阻塞调度。

> ⏳ Phase 2 可评估延迟消息替代 cron 扫描。Phase 1 保持 cron——触达对时间精度要求低（±1min 可接受）。

> 详见 [核心引擎规格 §1.2 Trigger-to-Event 线程隔离](./MOCASA催收系统升级_Phase1_核心引擎规格.md#12-trigger-to-event-线程隔离)；事件 Payload Schema 见 [领域模型与数据定义](./MOCASA催收系统升级_Phase1_领域模型与数据定义.md)。

#### 1.3.2 决策上下文快照化

**缺失后果**：并发消费时多个步骤同时实时查 DB，连接池成为吞吐瓶颈，高峰期 Consumer 线程相互阻塞。

案件接入时，数据接入层将用户画像核心标签序列化为不可变 JSON 快照写入 `t_contact_plan.context_snapshot`，SPI 决策实现读快照而非实时查 DB。不可快照的实时状态（还款/冻结）由 PreFlightChecker 和 ExecutionGuard 在步骤执行时实时校验。

> 详见 [核心引擎规格 §5.7 ExecutionContext](./MOCASA催收系统升级_Phase1_核心引擎规格.md#57-共享-dto-定义) / [§3.1 execute_step](./MOCASA催收系统升级_Phase1_核心引擎规格.md#31-executestep-七步管线)。

#### 1.3.3 幂等键契约

**缺失后果**：Redis Stream At-least-once 语义下，同一事件重复消费导致同一步骤被多次发送给用户。

每个步骤生成唯一 `idempotency_key`。核心引擎执行骨架首步通过分布式锁排他；渠道执行子层在调用供应商前通过 Redis SETNX 做二次去重。双层保障。

> 详见 [核心引擎规格 §3.1 步骤①](./MOCASA催收系统升级_Phase1_核心引擎规格.md#31-executestep-七步管线) / [§6.2 Redis 使用规范](./MOCASA催收系统升级_Phase1_核心引擎规格.md#62-redis-使用规范)。

#### 1.3.4 并发竞态控制

**缺失后果**：`PLAN_STEP_DUE`（Consumer A）与 `REPAYMENT_RECEIVED`（Consumer B）同时到达同一计划时，两个线程并发修改同一状态记录，导致"已还款的案件仍发出催收"或状态数据污染。幂等键解决的是"同一事件重复消费"，并发竞态控制解决的是"不同事件、同一目标、同一时刻"——两个独立的问题。

`EventConsumerDispatcher` 消费任何事件后，通过 `SELECT FOR UPDATE` 获取计划行级锁，修改完成后释放。持锁期间检查计划是否已进入终态——若是则静默退出，不执行任何业务操作。

> 详见 [核心引擎规格 §1.3 并发与一致性模型](./MOCASA催收系统升级_Phase1_核心引擎规格.md#13-并发与一致性模型)。

#### 1.3.5 事件总线可靠性保障

**缺失后果**：事件总线是所有模块的唯一通信介质。若消费循环静默假死（无错误日志、无告警），整个系统停止处理所有事件，触达任务全部积压。

`CollectionEventBus` 接口定义于共享模块，Phase 1 由基于 Lettuce 原生 API 的 `RedisStreamEventBusImpl` 实现（Spring Boot 2.1 无标准 Stream 抽象）。因手写消费循环存在连接假死风险，引入消费线程看门狗：循环每次轮询更新心跳时间戳，守护线程定期检查，超时则重建消费上下文并触发告警。接口抽象使未来替换消息中间件时业务代码零改动。

> 详见 [数据接入与事件规格 §事件总线](./MOCASA催收系统升级_Phase1_数据接入与事件规格.md)。

#### 1.3.6 死信队列

**缺失后果**：事件消费失败时，失败事件要么永久丢失（触达缺失），要么阻塞 Stream 后续消费（系统停摆），二者均不可接受。

三级恢复：即时重试（指数退避）→ 自动重放（MySQL 持久化 + 定时扫描重投）→ 人工介入。关键补充：重放前检查合规时段，避免"业务时间毒丸"——原始事件在合规窗口内触发，进入死信后在禁呼时段被盲目重放。

> 详见 [渠道编排规格 §错误恢复](./MOCASA催收系统升级_Phase1_渠道编排规格.md)。

#### 1.3.7 异步回调对账

**缺失后果**：事件驱动模型假设"事件终将到达"，但人工外呼的回调事件来自外部系统，存在永久丢失的可能。若无对账，计划静默卡死在 STEP_EXECUTING——无错误日志、无告警、无法自愈。

定时对账清理器扫描超时未回调的步骤，主动查询外部系统状态并补发事件，恢复状态机流转。

> 详见 [渠道编排规格 §15.2 兜底扫描任务](./MOCASA催收系统升级_Phase1_渠道编排规格.md#152-兜底扫描任务)。

### 1.4 渠道编排层

渠道编排是系统的完整出站链路，从业务决策到渠道发送形成闭环。物理上跨 `engine.strategy` + `collection-channel` 两个包，逻辑上是一个统一模块。

**内部分层**：

| 子层 | 职责 | 边界约束 |
|---|---|---|
| 策略子层 (`engine.strategy`) | 实现 5 个 SPI 接口：计划创建、合规校验、渠道决策、推进策略、穷尽处理 | 决策基于 context_snapshot（不可变），实时数据通过 ExecutionGuard 的 Redis 计数器获取 |
| 执行子层 (`collection-channel`) | 模板渲染、幂等校验、熔断降级、供应商 API 调用 | 哑管道原则：不查业务数据库，上下文由 StepCommand 传入 |

**执行子层核心能力**：

| 能力 | 说明 |
|---|---|
| 模板匹配与渲染 | 基于 StepCommand.templateHint 匹配最终模板，变量替换 |
| 渠道适配 | 统一 ChannelAdapter 接口，8 种渠道独立可启用/禁用 |
| 熔断与降级 | 按渠道+供应商粒度独立熔断（Redis 滑动窗口）；熔断后走 fallback 渠道；降级标记 `isFallback` 跳过合规计数 |
| 急停控制 | `t_system_property` 标志位支持全局暂停、单渠道禁用 |

**渠道分类**：

| 类别 | 渠道 | 特征 |
|---|---|---|
| 消息类 | SMS、Push、Email、Viber、WhatsApp | 同步返回，秒级完成 |
| 电话类 | AI_CALL、TTS (Text-to-Speech, 文字转语音播报) | 异步回调 |
| 人工类 | HUMAN_CALL | 批量预测式外呼，长生命周期异步会话 |

> 渠道编排内部设计、Adapter 规范、供应商错误码映射、人工外呼编排、合规检查链、规则匹配机制见 [渠道编排规格](./MOCASA催收系统升级_Phase1_渠道编排规格.md)。

### 1.5 应用层

| 职责 | 说明 |
|---|---|
| 管理后台 REST API | 案件详情、触达时间线、计划状态、合规记录可视化；Shiro 鉴权 |
| Webhook 回调入口 | 统一接收外部供应商回调，鉴权后发布 `CHANNEL_CALLBACK` 事件到事件总线 |
| XXL-Job 任务注册 | Trigger-to-Event 模式，仅发事件不跑业务逻辑，毫秒级返回 |
| 到期前通知接管 | D-3 ~ D0 的 Push/SMS 职责完整迁移至新系统 |

> 迁移方案见 [数据接入与事件规格 §迁移](./MOCASA催收系统升级_Phase1_数据接入与事件规格.md)。

## 2. 技术栈决策

| 维度 | 现状 | Phase 1 决策 | 理由 |
|---|---|---|---|
| 框架 | Spring Boot 2.1 (EOL) | **保持 2.1** | 框架升级作为独立后续项目 |
| 安全 | Shiro + 责任链 | **保持 Shiro** | 新模块不依赖，接口隔离 |
| 数据库 | MySQL + MyBatis | **保持 MyBatis + PageHelper** | 团队一致性 |
| 缓存 | Redis | **统一：缓存 + 限流 + 事件总线 + 幂等** | 合规计数器、context cache、idempotency_key |
| 内部事件 | 无 | **Redis Stream + CollectionEventBus 抽象** | 持久化 + ACK + Consumer Group；接口解耦实现 |
| 外部消息 | GCP PubSub (同步) | **异步并发消费** | 提升吞吐 |
| 调度 | XXL-Job | **Trigger-to-Event 模式** | 仅发事件，不执行业务 |
| 工作流 | 无 | **自建状态机 + SPI 模式** | Phase 1 需求可控；SPI 解耦策略与流程 |
| HTTP Client | RestTemplate | **RestTemplate + @Async** | 日均几百次调用，线程池足够 |
| 监控 | 无 | **Micrometer + Prometheus + Grafana** | 渠道健康 + 触达效果 + 放弃率 |

**框架升级路线图**：Phase 1 上线 2-3 个月后启动 Spring Boot 3.2+ / Java 17 / Spring Security 6.x / WebClient。

## 3. 扩展性与演进路径

### 3.1 容量扩展

**当前容量评估**：日均案件量 1w-2w，按每案件 3-5 步骤计算，日均步骤执行 3w-10w 次。集中在 8 小时业务窗口，峰值 QPS 约 1-3。当前单实例 + 8 线程 Consumer 池 + Redis Stream + MySQL 的组合有 10 倍以上余量，充分验证了 Phase 1 不做分布式拆分的决策。

Phase 1 单实例部署，架构关键路径均已参数化，扩容时改配置而非改代码：

| 瓶颈点 | 扩容方式 |
|---|---|
| 事件消费吞吐 | 新实例加入 Redis Stream Consumer Group |
| 渠道 I/O 并发 | 调整 Consumer executor-threads 线程池大小 |
| 定时扫表 | XXL-Job 开启分片广播模式 |
| 数据增长 | `t_contact_timeline` 按月分区，按需扩展 |

### 3.2 演进预留

SPI 架构下，Phase 2 演进只需新增实现类或替换注入配置：

| 演进方向 | Phase 1 SPI 接口 | Phase 2 新增实现 |
|---|---|---|
| LLM 决策引擎 | `DecisionEngine` (被 SPI 内部调用) | `LLMAgentDecisionEngine`，配置切换或 A/B |
| LLM 动态计划 | `PlanFactory` + `AdvancementPolicy` | LLM 生成计划 + 实时调整步骤 |
| LLM 话术生成 | `StepResolver`（内部调 TemplateService） | LLM 推荐最优话术 / 生成个性化内容 |
| 智能续建 | `ExhaustionPolicy` | LLM 判断是否值得继续 + 生成新策略 |
| 通话质检 | `TranscriptService` 接口（预留） | LLM 分析转写文本，自动评分 |
| 事件总线替换 | `CollectionEventBus` 抽象接口 | 替换实现类，业务代码零改动 |
| 微服务拆分 | 模块边界清晰，可独立编译 | 按需将 engine / channel 独立部署 |

**架构风险决策**：

| 风险 | 决策 | 理由 |
|---|---|---|
| Pre-flight 竞态空窗 | **Phase 1 接受** | 概率极低（空窗 < 500ms），后置补偿可覆盖。Phase 2 方向：Redis 临界标记 + 紧急拦截 Stream |
| Lettuce 连接假死 | **Phase 1 加固** | 看门狗机制成本极低且防灾难性停摆（详见 §1.3.5） |
| Webhook 回调丢失 | **Phase 1 加固** | 发生概率高，影响面大，对账清理器可控（详见 §1.3.7） |
| DLQ 合规时段碰撞 | **Phase 1 加固** | 成本极低，避免触达计划空跑（详见 §1.3.6） |
| AI_CALL/TTS 在途呼叫不可中止 | **Phase 1 接受** | 还款取消计划时无法终止已发起的 AI 外呼（LTH 不提供单次呼叫取消 API）；用户还款后仍可能接到一通催收电话。HUMAN_CALL 通过 `filterRepaidUser()` 覆盖。Phase 2 方向：评估 LTH 呼叫中止接口 |
| 跨计划取消原子性间隙 | **Phase 1 接受** | REPAYMENT_RECEIVED 逐个取消用户多个活跃计划，cancel(plan-A) 与 lock(plan-B) 之间存在窗口，plan-B 的步骤可能在此窗口内执行一次触达。概率低（用户通常单计划），后置补偿（timeline 记录 + 对账）可覆盖 |
